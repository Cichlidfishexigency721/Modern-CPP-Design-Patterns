# FSM: MODERN VARIANT IMPLEMENTATION (C++17/23)

## Intent
To implement a Finite State Machine using `std::variant` and `std::visit`
instead of traditional class inheritance. This version is functionally identical
to the GoF implementation (`02_CDPlayer_GoF`) but follows a completely different
architectural paradigm: **Value Semantics**.

## The OCP (Open/Closed Principle) Dilemma
It is crucial for students to understand that using `std::variant` changes the
extensibility axis of the Open/Closed Principle:

1. **GoF Approach (Traditional Polymorphism)**:
   - **Open for New States**: You can add a `FastForward` state by creating a new
   - **file and registering it without touching existing code.  Closed for New
   - **Events**: Adding a method like `powerConsumption()` requires updating the
   - **base interface and all existing state classes.

2. **Modern Variant Approach**:
   - **Closed for New States**: Adding a state requires modifying the
   - **`StateVariant` definition in `FSM.h`. It is "intrusive" regarding the
   - **state list.  Open for New Operations**: You can add new behaviors
   - **(Visitors) in separate files without touching the state structures
   - **themselves.

## Why use std::variant? (Performance & Safety)
If the GoF version is more "Open" for states, why does the industry prefer
Variants for critical systems?
   - **Value Semantics (Stack vs. Heap)**: In the GoF version, every state change
   - **usually involves `new` and `delete`. In the Variant version, the state lives
   - **entirely on the **Stack**. Changing states is as fast as copying a small
   - **structure.  Deterministic RAII**: We use constructors to turn lights ON
   - **(`onEntry`) and destructors to turn them OFF (`onExit`). By using
   - **`std::monostate` during transitions, C++ guarantees the old state is
   - **destroyed completely before the new one is built.  Static Safety
   - **(Exhaustiveness)**: `std::visit` forces the compiler to verify that every
   - **possible state is handled. If you add a state but forget to handle an event,
   - **the program will simply **fail to compile**.

## Conclusion
   - **Use the GoF Pattern (02)** for large, modular systems where third-party
   - **developers need to add their own states at runtime (plugins).  Use the
   - **Modern Variant (03)** for critical systems (embedded, real-time,
   - **high-performance) where the set of states is well-known and you prioritize
   - **memory safety and execution speed.

## Implementation Details
This example unifies the state logic into `States.cpp` to demonstrate the **high
cohesion** provided by the Variant approach.
