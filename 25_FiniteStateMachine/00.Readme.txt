# FINITE STATE MACHINE (BEHAVIORAL)

## Intent
The State pattern allows an object to alter its behavior when its internal 
state changes. The object will appear to change its class.

## The Problem
When an object has many states and its behavior changes drastically depending 
on the current state, a standard implementation would involve a massive amount 
of 'if-else' or 'switch-case' statements scattered throughout the class. 
This is not only hard to maintain but also error-prone, as adding a new 
state requires modifying every single conditional branch.

## The Solution
The pattern suggests moving the state-specific behavior out of the Context 
class into a set of separate "State" classes. 
- **Context:** Defines the interface for clients and maintains a reference to 
the current State object.
- **State:** Defines the interface for encapsulating behavior associated with 
a particular state.
- **Concrete State:** Each subclass implements the behavior associated with a 
specific state of the Context.

## Our Examples
This folder illustrates the evolution of the State pattern across different 
architectural paradigms:

1. **01_SimpleExample/:**
A basic theoretical implementation showing the "handshake" between a 
Context (FSM) and its States (State_A, State_B). It demonstrates how 
transition actions and entry/exit behaviors are triggered during state changes.

2. **02_CDPlayer_GoF/:**
A realistic scenario simulating a CD Player using the classic Gang of Four 
approach. It showcases how complex systems can be managed by encapsulating 
logic in separate files for each state, avoiding "God Objects" and making the 
logic modular.

3. **03_CDPlayer_Variant/:**
A modern C++17/23 implementation of the same CD Player using std::variant 
and std::visit. This version moves away from inheritance and pointers, 
embracing **Value Semantics**. It offers higher performance (Stack vs Heap) 
and static safety (the compiler ensures all states are handled via 
exhaustiveness checking).

## Key Benefits
- **Single Responsibility Principle:** State-specific logic is localized in 
its own class or structure.
- **Open/Closed Principle:** You can introduce new states (in GoF) or new 
operations (in Variant) with minimal impact on existing code.
- **Complexity Reduction:** Eliminates large conditional statements, making 
the system flow easier to understand and debug.

## NOTE ON AUTO-REGISTRATION TECHNIQUES
In our GoF-based implementations, we use a technique to register states 
automatically when the program starts. This is done using a static 
lambda-based initialization:

```cpp
static inline bool registered_ = []() {
   FSM:registerState<T>("Name");
   return true;
}();
```

This is preferred for two reasons:
- **Memory Efficiency:** We only store a 'bool' (1 byte) in the static segment, 
instead of instantiating the entire state object in the heap or static 
memory before the main starts.
- **Encapsulation:** The registration logic is defined outside the class 
definition, ensuring the compiler has full knowledge of the type 'T' 
before it attempts to instantiate it, thus avoiding "incomplete type" 
errors during compilation.
