# COMMAND PATTERN (BEHAVIORAL)

## Intent
Encapsulate a request as an object, thereby letting you parameterize clients 
with different requests, queue or log requests, and support undoable 
operations.

## The Problem
In many applications, we need to issue requests to objects without knowing 
anything about the operation being requested or the receiver of the request. 
Hardcoding these calls leads to tight coupling.

## The Solution
The Command pattern turns a request into a stand-alone object that contains 
all information about the request (the receiver, the method to call, and 
any parameters). This transformation lets you:
1. Pass commands as method arguments.
2. Store commands in a collection (queue, undo stack).
3. Execute commands at a later time (deferred execution).

## Our Examples
This directory illustrates the evolution of the Command pattern from the 
classic object-oriented approach to modern functional-style C++:

1. **01_Command_GoF/:**
   The traditional implementation using class inheritance. Each command is 
   a class that implements a virtual 'execute()' method. It uses 
   'std::unique_ptr' to manage the lifecycle of commands in a queue.

2. **02_Modern_Variant/:**
   A modern C++17 approach. Commands are simple data structures (structs), 
   and the execution logic is centralized in a Visitor. This version is 
   "non-intrusive" and uses value semantics (no pointers/heap).

## Key Benefits
- **Decoupling:** The Invoker is separated from the Receiver.
- **Extensibility:** Adding new commands doesn't require changing existing code.
- **Composability:** Commands can be assembled into 'MacroCommands'.
