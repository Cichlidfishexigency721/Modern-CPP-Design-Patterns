# ERROR HANDLING PATTERNS (RESULT / EITHER)

## Intent
To represent the outcome of an operation that might fail by returning a single
object that encapsulates either the successful result or a detailed error
description. This approach avoids the overhead of exceptions and the ambiguity
of traditional error codes.

## The Problem
In traditional C++, we have two main ways to handle failures, both with significant drawbacks:
1. **Error Codes (C-style):** Usually an 'int' returned by the function.
   - Drawback: They are easily ignored by developers.
   - Drawback: They force the use of output parameters (e.g., passing a
               pointer to get the actual result), which makes the API clunky.
2. **Exceptions:**
   - Drawback: They introduce a hidden control flow that is hard to follow.
   - Drawback: They have a performance cost due to "stack unwinding" when an
               error actually occurs.

## The Solution: "Errors as Values"
Modern C++ leans towards functional-style error handling. Instead of throwing an
exception, the function returns a "Sum Type" that can hold either the value or
the error.

Example of the modern syntax:
```cpp
// Instead of: double result = calculate(input); (might throw)
// We use: auto result = safe_calculate(input);
if (result) { use_value(*result); }
else        { handle_error(result.error()); }
```

## Our Examples
This directory demonstrates the transition from a manual implementation to the
official C++ standard:
1. **01_ErrorHandling_Variant/ (C++17):**
   Uses std::variant<Result, Error> to manually implement the "Either"
   pattern. It is the foundation of modern error handling and works on any
   system with C++17 support.

2. **02_ErrorHandling_Expected/ (C++23):**
   Uses the new std::expected utility. This is the industrial standard
   introduced in C++23, providing a much more ergonomic and readable API
   specifically designed for this purpose.

## Key Benefits
- **Safety:** The programmer is explicitly reminded (by the type system) that the
              operation can fail.
- **Clarity:** No more output parameters; the return type tells the whole story.
- **Efficiency:** It uses standard control flow (if/else), making it faster than
                  exceptions in error-heavy scenarios.
